# StudyZone（学习园地）整体架构设计

> 版本：v1.0 ｜ 编写日期：2026-05-26
> 目标：复刻多邻国式游戏化学习体验，定位为支持多学科（语言 / 数学 / 音乐 等）的通用学习平台，Web + 移动端双形态，面向真实上线运营。

---

## 一、产品定位与目标

**一句话定义**：StudyZone 是一款以"短时高频 + 游戏化激励"为核心交互的多学科学习平台，用户每天通过 5–15 分钟的关卡式练习持续积累技能。

**核心价值主张**

1. **低门槛高粘性**：把学习内容拆解为 30 秒一题的小步骤，配合连胜（Streak）、经验值（XP）、宝石（Gem）等机制提升留存。
2. **多学科可扩展**：底层"技能树 + 关卡 + 题型"框架与具体内容解耦，初期可上线英语、日语，后续按相同模型扩展数学、音乐等。
3. **社交驱动**：好友、周联赛（League）、好友 PK 让学习从单机变成有竞争感的社区行为。
4. **全平台一致**：Web 端用于桌面学习与课程编辑，移动端聚焦碎片时间练习，数据与进度全平台同步。

**北极星指标候选**：DAU × 日均完成关卡数 × 7 日留存率。

**非目标（v1 不做）**：直播课、1v1 真人辅导、UGC 课程市场、本地缓存离线学习（v2 再加）。

## 二、用户角色与核心场景

| 角色 | 描述 | 关键场景 |
|---|---|---|
| 学习者（C 端核心） | 普通用户，注册后选择课程开始学习 | 每日打卡、闯关、复习、好友 PK、加入联赛 |
| 内容编辑（B 端） | 内部教研 / 外包教师，制作课程内容 | 在 CMS 后台编辑技能树、关卡、题目，灰度发布 |
| 运营 / 增长 | 运营人员 | 推送通知、活动配置、数据看板、A/B 测试 |
| 管理员 | 平台管理者 | 用户管理、权限、风控、内容审核 |

**学习者核心闭环**

```
注册/登录 → 选择课程语言 → 进入技能树
       ↓
   选择当前关卡 → 进入答题流 → 完成关卡获得 XP/宝石
       ↓
   更新连胜 → 触发段位/联赛排名变化 → 推送好友动态
       ↓
   次日推送提醒 → 回到 App 继续学习
```

## 三、核心功能模块

整个产品按"领域"拆分为 8 个一级模块，每个模块对应后端的一个 Bounded Context（领域边界）。

### 3.1 账户与身份（Account）
- 邮箱 / 手机号 / 第三方（Google / Apple / 微信）登录
- JWT + Refresh Token 鉴权
- 个人资料、头像、学习目标设定、通知偏好
- 账户安全：二次验证、风控、设备管理

### 3.2 课程与内容（Curriculum）
- 学科（Subject）→ 课程（Course）→ 单元（Unit）→ 技能（Skill）→ 关卡（Lesson）→ 题目（Exercise）
- 题型库（多选、文本输入、听力、口语、配对、排序、看图选词…）
- 内容版本控制（草稿 / 灰度 / 正式发布）
- 国际化（i18n）支持多语言界面与题面

### 3.3 学习引擎（Learning Engine）
- 关卡运行时（Session）：抽题、判分、错题入队
- 间隔复习（SRS, Spaced Repetition）：基于遗忘曲线推送复习关卡
- 自适应难度：根据用户掌握度调整出题概率
- 心数（Hearts / 体力）机制：错题扣心，引导购买宝石或等待恢复

### 3.4 游戏化（Gamification）
- XP（经验值）+ 等级
- 连胜（Streak）+ 连胜冻结道具
- 宝石（Gem）虚拟货币 + 商店
- 成就 / 徽章
- 每日 / 每周任务

### 3.5 社交（Social）
- 好友系统（搜索、添加、关注）
- 排行榜：好友榜 / 全国榜 / 周联赛（升降级机制）
- 好友 PK / 双人对战
- 社区动态（可选 v2）

### 3.6 通知与触达（Notification）
- App 推送（FCM / APNs / 极光）
- 站内消息中心
- 邮件 / 短信（重要事件）
- 智能提醒：基于学习时间习惯的个性化提醒

### 3.7 支付与会员（Billing）
- 订阅会员（去广告、无限心数、专属功能）
- 一次性消费（宝石包）
- 多渠道支付：Apple IAP / Google Play / 支付宝 / 微信 / Stripe
- 订单、发票、退款、对账

### 3.8 后台与运营（Admin / Ops）
- CMS：课程内容编辑器（树状结构编辑、题目预览）
- 用户管理、权限管理（RBAC）
- 数据看板（埋点、漏斗、留存）
- A/B 实验配置
- 客服工单

## 四、系统总体架构

整体采用 **「模块化单体 → 渐进微服务」** 演进策略：MVP 阶段一个后端服务即可，但代码按 Bounded Context 严格分包，未来按流量切分微服务零阻力。

### 4.2 关键架构决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 后端形态 | 模块化单体（Nest.js Monorepo），预留微服务拆分点 | MVP 快、易维护；DDD 分包让未来拆分零成本 |
| 前端形态 | Web 用 Next.js（App Router + RSC），App 用 React Native + Expo | 同一套 React 心智模型，组件 / 业务逻辑可在 packages/shared 复用 |
| 数据库 | PostgreSQL 主库 + Redis 缓存 + S3 对象存储 | Postgres 关系建模强，JSONB 支撑题目结构灵活性 |
| 异步通信 | Kafka（或 Redis Stream） + BullMQ 任务队列 | 解耦学习事件、推送、积分结算等 |
| 实时通信 | WebSocket（Socket.IO） | 联赛榜、好友 PK 实时同步 |
| 鉴权 | OAuth2 + JWT（短期） + Refresh Token（长期，存 Redis） | 行业标准、移动端友好 |
| 部署 | Docker + Kubernetes（生产）/ Docker Compose（本地） | 上线产品的标准实践 |
| 多端代码共享 | TypeScript Monorepo（pnpm + Turborepo） | 类型在端到端打通 |

## 五、技术栈选型

### 5.1 客户端

**Web 端 — Next.js 14**
- App Router + React Server Components：营销页 / 落地页 SEO 友好
- TanStack Query：服务端状态管理
- Zustand：客户端轻量状态
- Tailwind CSS + shadcn/ui：组件库与设计系统
- Framer Motion：游戏化动画（升级、连胜、宝箱）
- next-intl：国际化

**移动端 — React Native + Expo**
- Expo Router：与 Next.js 心智一致
- React Native Reanimated + Skia：高性能动画与画布
- React Native MMKV：本地存储
- Expo Notifications：推送
- EAS Build：云端打包

**Admin CMS — React + Vite**
- 独立 SPA，给内容编辑团队用
- Ant Design Pro 或 Refine 框架
- 富文本与树形结构编辑器（基于 dnd-kit）

### 5.2 服务端

| 组件 | 选型 | 说明 |
|---|---|---|
| 后端框架 | NestJS（TypeScript） | DDD 模块化，依赖注入，生态完善 |
| ORM | Prisma | 类型安全，迁移工具好 |
| API 风格 | REST（主） + GraphQL（Admin CMS） + WebSocket（实时） | 各取所长 |
| 任务队列 | BullMQ（Redis 驱动） | 复习提醒、推送、对账等 |
| 消息总线 | Kafka 或 Redis Stream | 跨模块事件（如「关卡完成」事件触发积分、徽章、推送） |
| 搜索 | Elasticsearch / Meilisearch | 课程搜索、用户搜索 |
| 文件存储 | S3 / 阿里云 OSS | 题目音频、用户头像 |
| 媒体处理 | FFmpeg（Worker） | 听力题音频转码、口语题录音处理 |
| AI 能力（可选） | OpenAI / Claude API | 口语评分、AI 助教、内容生成草稿 |

### 5.3 数据与基础设施

| 类别 | 选型 |
|---|---|
| 主数据库 | PostgreSQL 15（RDS / 阿里云 RDS） |

## 七、API 设计风格

### 7.1 REST 资源约定

```
POST   /api/v1/auth/login
POST   /api/v1/auth/register
POST   /api/v1/auth/refresh

GET    /api/v1/me
PATCH  /api/v1/me

GET    /api/v1/courses
POST   /api/v1/courses/:id/enroll
GET    /api/v1/courses/:id/tree             -- 用户视角的技能树

GET    /api/v1/lessons/:id                  -- 开始关卡（返回题目序列 + sessionId）
POST   /api/v1/sessions/:id/attempts        -- 提交单题作答
POST   /api/v1/sessions/:id/complete        -- 关卡结算

GET    /api/v1/leagues/me                   -- 当前所在联赛
GET    /api/v1/friends
POST   /api/v1/friends/requests

POST   /api/v1/billing/checkout
POST   /api/v1/billing/webhook              -- 各支付渠道回调
```

约定：

- 全部 `application/json`，时间用 ISO 8601 + UTC。
- 错误结构 `{ code, message, details? }`；HTTP 状态码语义化。
- 写操作要求 `Idempotency-Key` 头（特别是支付）。
- 分页用 `cursor + limit`，避免深分页性能问题。

### 7.2 GraphQL（Admin CMS）

- 资源复杂、关联多，Admin 用 GraphQL 一把梭。
- 客户端用 Apollo / urql。
- 严格区分 `Public Schema`（C 端） 与 `Admin Schema`，避免越权。

### 7.3 实时通道（WebSocket）

```
WSS /ws  --  鉴权握手
事件：
  league.rank.updated
  friend.activity
  pk.invite / pk.move / pk.result
  notification.new
```

### 7.4 事件总线（内部）

跨模块解耦事件示例：

```
learning.lesson.completed
  payload: { userId, lessonId, xp, errors }
  消费者：Gamification（结算 XP/streak）、Notification（推送）、Analytics
```

  │                                              · 朋友推送        │

### 8.2 每日学习提醒

```
定时任务（Cron, 23:00 用户本地时间）
  → 查询今天未学习且 streak > 0 的用户
  → 进入 BullMQ 推送队列
  → Worker 调用 FCM/APNs
  → 落 PushLog
```

## 十、开发路线图（建议）

按"先跑通主链路，再加深"的原则分四期。

### Phase 0 — 基础设施搭建（2 周）
- Monorepo 脚手架（pnpm + Turborepo + ESLint + Prettier + Husky）
- Docker Compose 本地开发栈（PG + Redis + MinIO）
- CI/CD 流水线（GitHub Actions：lint / test / build / preview deploy）
- 设计系统：Tailwind 主题 + shadcn/ui + 移动端 Tamagui

### Phase 1 — MVP 主干（6–8 周）
**目标：单人能从注册到完成一节英语课**
- 账户：邮箱注册登录、个人资料
- 课程：硬编码一门"中文用户学英语"课程（10 个技能、每个 3 关）
- 题型：先实现 4 种（多选、文本输入、配对、听力）
- 学习引擎：关卡运行、判分、XP 结算
- 游戏化：XP、连胜、心数
- Web 端 + 移动端 UI 双端打通

### Phase 2 — 留存与社交（4–6 周）
- SRS 间隔复习
- 好友系统 + 周联赛
- 推送通知 + 每日任务
- 排行榜实时更新（WebSocket）
- 数据埋点 + 留存看板

### Phase 3 — 商业化与扩科（6 周）
- 订阅会员（Stripe + Apple IAP + Google Play）
- 宝石商店
- 第二个学科：数学（题型复用 80%）
- CMS 内容编辑器
- A/B 实验框架

### Phase 4 — 规模化（持续）
- 微服务拆分（先拆 Learning / Social）
- 多区域部署
- AI 助教（口语评分、个性化题目）
- 离线学习模式

## 十二、风险与开放问题

1. **内容生产是瓶颈**：题库质量决定产品上限，需尽早搭好 CMS 与教研工作流。
2. **跨端动画一致性**：Web 用 Framer Motion，RN 用 Reanimated，需要建立设计语言文档保证体验一致。
3. **支付合规**：iOS 上虚拟商品必须走 Apple IAP，分成 30%，定价策略需早期规划。
4. **AI 集成的成本**：口语评分若用第三方 API，单次成本可能高于 XP 收益，需评估自研 / 缓存方案。
5. **冷启动留存**：游戏化机制是否真正生效，需在 Phase 1 末期就上埋点验证，而非等到 Phase 2。
